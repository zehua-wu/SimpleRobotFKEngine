# MiniRobotFKLab

MiniRobotFKLab is a small hands-on robotics project for understanding how robot models, kinematic trees, and forward kinematics work under the hood.

The goal is not to build a full physics simulator. Instead, this project focuses on the core idea behind robot modeling packages such as MuJoCo, Drake, and Pinocchio:

> A robot is a tree of links connected by joints.
> Given the joint values, each link pose can be updated recursively from the root to the leaves.

This project connects three ideas:

1. Computer graphics scene graphs
2. Robot kinematic trees
3. Forward kinematics in robotics simulators

---

## Project Goals

This project aims to:

* Build simple robot models manually using MJCF/XML or URDF
* Load and visualize these models in MuJoCo
* Change joint values and observe how the whole robot pose updates
* Implement a minimal kinematic tree update system from scratch
* Implement a simple forward kinematics engine
* Compare the custom FK results with MuJoCo's computed body/site poses
* Understand the relationship between links, joints, transforms, meshes, and visualization

---

## Motivation

Modern robotics packages already provide strong infrastructure for modeling, visualization, kinematics, dynamics, contact, and simulation. However, it is still important to understand what happens underneath a simple call such as:

```python
mujoco.mj_forward(model, data)
```

or

```python
plant.CalcRelativeTransform(context, frame_A, frame_B)
```

At the forward kinematics level, the core mechanism is similar to a computer graphics scene graph:

```text
parent world transform
        ×
child local transform
        =
child world transform
```

For a robot, the child local transform is not arbitrary. It is generated by a joint:

```text
child local transform =
    joint origin transform
    × joint motion transform(q)
```

This project implements this idea in a minimal and understandable way.

## Core Concepts

### Scene Graph

In computer graphics, a scene graph is a hierarchy of nodes. Each node has a local transform relative to its parent.

```text
world
 └── body
      ├── left_arm
      │    └── left_hand
      └── head
```

The world transform of each child node is computed recursively:

```text
T_world_child = T_world_parent @ T_parent_child
```

### Robot Kinematic Tree

A robot model is similar, but nodes are links and edges are joints.

```text
base_link
 └── joint1 → link1
      └── joint2 → link2
           └── joint3 → end_effector
```

For each joint:

```text
T_world_child =
    T_world_parent
    @ T_joint_origin
    @ T_joint_motion(q)
```

### Mesh and Geometry

A mesh or primitive geometry is attached to a link.

The link determines where the object is in the world.
The mesh determines what the object looks like.

```text
link
 ├── visual geometry
 ├── collision geometry
 └── inertial properties
```

In this project, simple geometries such as boxes, capsules, cylinders, and spheres are used instead of realistic CAD meshes.

### 2. Toy 6-DOF Robot Arm

A simple serial robot arm with six revolute joints.

```text
base
 └── shoulder_yaw
      └── shoulder_pitch
           └── elbow_pitch
                └── wrist_roll
                     └── wrist_pitch
                          └── wrist_yaw
                               └── end_effector
```

This model is used to understand how a serial chain propagates joint motion from the base to the end-effector.

---

### 3. Toy Quadruped

A simplified quadruped with a trunk and four legs.

```text
trunk
 ├── front_left_leg
 ├── front_right_leg
 ├── rear_left_leg
 └── rear_right_leg
```

Each leg starts with 2 or 3 revolute joints:

```text
hip_pitch
knee_pitch
```

or

```text
hip_roll
hip_pitch
knee_pitch
```

This model is used to understand the difference between a serial chain and a branching kinematic tree.

## Implementation Plan

### Stage 1: Build a Minimal Scene Graph

Implement a generic node hierarchy:

```python
class Node:
    def __init__(self, name, local_T):
        self.name = name
        self.local_T = local_T
        self.world_T = np.eye(4)
        self.children = []
```

Recursive update rule:

```python
def update_tree(node):
    for child in node.children:
        child.world_T = node.world_T @ child.local_T
        update_tree(child)
```

This stage connects the project to computer graphics scene graph logic.

---

### Stage 2: Add Robot Joints

Extend the node system into a robot kinematic tree.

Each joint stores:

```text
parent link
child link
joint type
joint origin transform
joint axis
joint value q
```

Supported joint types:

* fixed
* revolute
* prismatic

The local transform is generated by:

```text
T_parent_child(q) =
    T_joint_origin
    @ T_joint_motion(q)
```

---

### Stage 3: Implement Forward Kinematics

Implement a minimal FK function:

```python
def forward_kinematics(robot, q):
    robot.root.world_T = np.eye(4)

    for joint in robot.joints_in_topological_order:
        parent = robot.links[joint.parent]
        child = robot.links[joint.child]

        T_origin = joint.origin_T
        T_motion = joint.motion_transform(q[joint.name])

        child.world_T = parent.world_T @ T_origin @ T_motion

    return {name: link.world_T for name, link in robot.links.items()}
```

This is the core of the project.

---

### Stage 4: Visualize Models in MuJoCo

Load the toy robot models into MuJoCo:

```python
import mujoco
import mujoco.viewer

model = mujoco.MjModel.from_xml_path("models/two_link_arm.xml")
data = mujoco.MjData(model)

with mujoco.viewer.launch_passive(model, data) as viewer:
    while viewer.is_running():
        mujoco.mj_forward(model, data)
        viewer.sync()
```

Change joint values manually:

```python
data.qpos[0] = 0.5
data.qpos[1] = -0.8
mujoco.mj_forward(model, data)
```

Observe how the body and end-effector poses update.

---

### Stage 5: Compare Custom FK with MuJoCo

For the same joint configuration:

1. Run custom FK
2. Run MuJoCo `mj_forward`
3. Compare body or site poses

Example comparison:

```python
p_custom = T_custom["link2"][:3, 3]
p_mujoco = data.body("link2").xpos

position_error = np.linalg.norm(p_custom - p_mujoco)
print("Position error:", position_error)
```

The expected error should be close to numerical precision if the transform conventions are consistent.

## Summary

MiniRobotFKLab is a small project for understanding the foundation of robot modeling and forward kinematics.

The central idea is simple:

```text
robot model + joint values
        ↓
recursive kinematic tree update
        ↓
world pose of every link and end-effector
```

This is the same core idea behind many robotics tools. MuJoCo, Drake, and Pinocchio provide much more powerful and optimized implementations, but the basic FK update logic can be understood and reproduced with a small amount of Python code.
